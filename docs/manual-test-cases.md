# Ручные тест-кейсы Control Habits

Документ для ручной проверки приложения тестировщиком. По каждому кейсу фиксировать результат: **OK** или **FAIL** (при FAIL — краткое описание ошибки для разработчика).

**Контекст:** веб — вход через Telegram (код привязки), редактор расписания, настройки, отчёт; бот — /start, пуши по расписанию, hotkey-кнопки, /active и «Закончить». API авторизуется заголовком `X-Web-Session-Id` после привязки.

---

## 1. Онбординг и привязка

### TC-AUTH-01 — Веб: кнопка «Войти через Telegram» и получение кода

**Предусловия:** Открыт сайт приложения (веб-клиент), бот и API доступны.

**Шаги:**
1. Открыть главную страницу (страницу входа).
2. Нажать кнопку «Войти через Telegram».

**Ожидаемый результат:** Появляется блок с кодом привязки (короткая строка из букв/цифр) и ссылкой вида «Открыть бота» (`t.me/<bot>?start=<code>`). Указано, что код действителен ограниченное время (например, 10 минут).

**Примечания:** Код должен быть пригоден для deep link (A–Z, a–z, 0–9, `_`, `-`).

---

### TC-AUTH-02 — Веб: копирование кода и переход по ссылке

**Предусловия:** На экране отображён код и ссылка (после TC-AUTH-01).

**Шаги:**
1. Нажать «Copy» (или аналог) у кода.
2. Проверить, что в буфере обмена — тот же код.
3. Нажать ссылку «Открыть бота» (или вставить код вручную в `t.me/<bot>?start=<code>` и открыть).

**Ожидаемый результат:** Открывается Telegram с ботом; бот получает /start с кодом и обрабатывает его (см. TC-AUTH-05).

---

### TC-AUTH-03 — Бот: /start без кода (приветствие)

**Предусловия:** Бот запущен. Пользователь ещё не привязан (или привязка не важна для текста).

**Шаги:**
1. Открыть бота по ссылке без параметра или написать `/start` без аргументов.

**Ожидаемый результат:** Бот отвечает приветствием: приглашение зайти на сайт, нажать «Войти через Telegram» и открыть бота по ссылке с кодом. URL сайта корректен (из конфига). Кнопка «Открыть сайт» опциональна.

---

### TC-AUTH-04 — Бот: /start с несуществующим кодом

**Предусловия:** Бот запущен.

**Шаги:**
1. Отправить боту команду `/start` с произвольным несуществующим кодом (например, `/start NOCODE123`).

**Ожидаемый результат:** Сообщение вида «Код не найден. Запроси код на сайте и открой меня по ссылке с кодом.»

---

### TC-AUTH-05 — Бот: /start с кодом — успешная привязка

**Предусловия:** На веб-странице получен код привязки (TC-AUTH-01). Код не истёк и не использован.

**Шаги:**
1. Открыть бота по ссылке `t.me/<bot>?start=<code>` (или ввести `/start <code>`).
2. Убедиться, что в веб-клиенте настроен polling или обновление по статусу привязки (если реализовано).

**Ожидаемый результат:** Бот отвечает: «Готово! Твой аккаунт привязан к сайту. Можешь вернуться в браузер и настроить расписание и кнопки.» После привязки веб-клиент (при наличии polling/redirect) переходит в настройки или на главный экран авторизованного пользователя.

---

### TC-AUTH-06 — Бот: /start с истёкшим кодом

**Предусловия:** Есть возможность получить код и дождаться истечения TTL (или вручную выставить в БД `expires_at` в прошлое для теста).

**Шаги:**
1. Получить код на сайте, дождаться истечения срока действия кода (или использовать код с истёкшим TTL).
2. Открыть бота по ссылке с этим кодом (`/start <code>`).

**Ожидаемый результат:** Бот отвечает: «Код устарел. Запроси новый код на сайте и открой меня снова по новой ссылке.» Привязка не создаётся и не обновляется.

---

### TC-AUTH-07 — Бот: /start с уже использованным кодом

**Предусловия:** Код уже был один раз успешно использован для привязки (выполнен TC-AUTH-05 с этим кодом).

**Шаги:**
1. Повторно открыть бота с тем же кодом (`/start <code>`), например с другого аккаунта Telegram или того же.

**Ожидаемый результат:** Бот отвечает: «Этот код уже использован. Если нужно привязать другой аккаунт — запроси новый код на сайте.» Новая привязка по этому коду не создаётся.

---

### TC-AUTH-08 — API: проверка статуса привязки по коду (GET link-status)

**Предусловия:** Доступ к API (curl/Postman или веб-клиент).

**Шаги:**
1. Создать код: `POST /api/auth/link-code` (тело пустое или с `web_session_id`).
2. Вызвать `GET /api/auth/link-status?code=<полученный_код>`.

**Ожидаемый результат:** В ответе `status: "pending"` до использования кода в боте. После успешного `/start <code>` в боте для того же кода (если проверка по коду поддерживается) — `consumed`; для истёкшего — `expired`; для несуществующего — `not_found`.

**Примечания:** В API статус по коду может не возвращаться после consume — тогда проверять по `web_session_id` (см. TC-AUTH-09).

---

### TC-AUTH-09 — API: проверка статуса привязки по web_session_id

**Предусловия:** Создан код с известным `web_session_id` (из ответа `POST /api/auth/link-code`).

**Шаги:**
1. До привязки: `GET /api/auth/link-status?web_session_id=<id>`.
2. В боте выполнить `/start <code>` для этого кода.
3. Снова `GET /api/auth/link-status?web_session_id=<id>`.

**Ожидаемый результат:** До привязки — `pending` (или `not_found` по реализации). После успешной привязки — `consumed`. В запросе передаётся только один параметр: либо `code`, либо `web_session_id` (иначе API возвращает 400).

---

### TC-AUTH-10 — Веб: переход в настройки после привязки

**Предусловия:** Реализован переход после привязки (polling link-status или redirect).

**Шаги:**
1. На сайте нажать «Войти через Telegram», получить код и ссылку.
2. Открыть бота по ссылке и завершить привязку.
3. Вернуться в браузер (при необходимости обновить страницу).

**Ожидаемый результат:** Пользователь оказывается в авторизованном состоянии: доступны разделы «Расписание», «Быстрые кнопки», «Уведомления», «Отчёт» (или эквиваленты). Без повторного ввода кода.

## 2. Веб: расписание

### TC-SCHED-01 — Создание шаблона расписания

**Предусловия:** Пользователь авторизован (заголовок `X-Web-Session-Id` с привязанной сессией). Шаблон ещё не создан (или допустимо создать новый).

**Шаги:**
1. Открыть раздел «Расписание» (редактор расписания).
2. Если шаблона нет — нажать создание шаблона / «Добавить» (согласно UI).
3. Ввести название шаблона и сохранить.

**Ожидаемый результат:** Шаблон создаётся, отображается в интерфейсе. API: `POST /api/schedule/template` с телом `{"name": "..."}` возвращает 201 и шаблон с `id`.

---

### TC-SCHED-02 — Выбор часового пояса

**Предусловия:** Пользователь авторизован. Открыт редактор расписания.

**Шаги:**
1. Найти блок выбора часового пояса (например, «Часовой пояс: [ Europe/Moscow ▼ ]»).
2. Выбрать другой пояс (например, `Europe/Moscow` или `Asia/Yekaterinburg`) и сохранить (если есть отдельная кнопка сохранения для настроек пользователя).

**Ожидаемый результат:** Часовой пояс сохраняется. При следующей загрузке отображается выбранное значение. API: `PATCH /api/users/me` с `{"timezone": "Europe/Moscow"}` возвращает обновлённые настройки; `GET /api/users/me` — актуальный timezone.

---

### TC-SCHED-03 — Добавление дела (TaskItem)

**Предусловия:** Есть шаблон расписания. Открыт редактор расписания.

**Шаги:**
1. Нажать «Добавить» (элемент плана).
2. Выбрать тип «Дело».
3. Ввести название (например, «Подъём»), время (например, 07:00), отметить дни недели (например, Пн–Пт).
4. Сохранить.

**Ожидаемый результат:** В списке расписания появляется элемент: одно время, название, тип «Дело». API: `POST /api/schedule/template/{template_id}/items` с `kind: "task"`, `start_time`, `end_time` (можно равным `start_time`), `days_of_week: [1,2,3,4,5]` — 201 и объект элемента.

---

### TC-SCHED-04 — Добавление события (EventItem)

**Предусловия:** Есть шаблон расписания.

**Шаги:**
1. Нажать «Добавить».
2. Выбрать тип «Событие».
3. Ввести название (например, «Учёба»), время начала (09:00), время конца (11:00), дни недели.
4. Сохранить.

**Ожидаемый результат:** В списке появляется элемент с диапазоном времени (09:00–11:00), тип «Событие». API: `POST .../items` с `kind: "event"`, `start_time`, `end_time`, `days_of_week` — 201.

---

### TC-SCHED-05 — Редактирование элемента плана

**Предусловия:** В шаблоне есть хотя бы один элемент (дело или событие).

**Шаги:**
1. Открыть редактирование элемента (меню ⋮ или двойной клик).
2. Изменить название и/или время и/или дни недели.
3. Сохранить.

**Ожидаемый результат:** Изменения отображаются в списке. API: `PATCH /api/schedule/plan-items/{plan_item_id}` с частичными полями возвращает обновлённый элемент.

---

### TC-SCHED-06 — Удаление элемента плана

**Предусловия:** В шаблоне есть элемент плана.

**Шаги:**
1. Выбрать удаление элемента (меню ⋮ → Удалить или аналог).
2. Подтвердить при необходимости.

**Ожидаемый результат:** Элемент исчезает из списка. API: `DELETE /api/schedule/plan-items/{plan_item_id}` — 204.

---

### TC-SCHED-07 — Удаление шаблона (опционально)

**Предусловия:** Есть шаблон (приложение допускает удаление шаблона в MVP).

**Шаги:**
1. Вызвать удаление шаблона из UI или API: `DELETE /api/schedule/template/{template_id}`.

**Ожидаемый результат:** Шаблон и связанные элементы плана удаляются (или возвращается 403/404 по правилам доступа). При отсутствии функции удаления шаблона — кейс пропустить.

---

## 3. Веб: активности и hotkey

### TC-ACT-01 — Создание активности (hotkey)

**Предусловия:** Пользователь авторизован.

**Шаги:**
1. Открыть раздел активностей / «Быстрые кнопки» (или каталог активностей).
2. Создать новую активность: название (например, «YouTube»), тип «hotkey».
3. Сохранить.

**Ожидаемый результат:** Активность появляется в списке с типом hotkey. API: `POST /api/activities` с `{"name": "YouTube", "kind": "hotkey"}` — 201 и объект с `id`.

---

### TC-ACT-02 — Создание активности (regular)

**Предусловия:** Пользователь авторизован.

**Шаги:**
1. Создать активность с типом «regular» (название любое).
2. Сохранить.

**Ожидаемый результат:** Активность создаётся, в списке отображается как обычная (не hotkey). В боте не появляется как быстрая кнопка.

---

### TC-ACT-03 — Добавление hotkey-кнопки (привязка активности к кнопке с подписью)

**Предусловия:** Есть минимум одна активность с типом hotkey.

**Шаги:**
1. В разделе «Быстрые кнопки в боте» нажать «Добавить».
2. Выбрать активность из списка и задать подпись на кнопке (например, «YouTube» или «Чтение»).
3. Сохранить.

**Ожидаемый результат:** В таблице быстрых кнопок появляется строка: активность, подпись, порядок. API: `POST /api/activities/hotkeys` с `{"activity_id": <id>, "label": "YouTube"}` — 201.

---

### TC-ACT-04 — Изменение порядка hotkey-кнопок

**Предусловия:** Настроено не менее двух hotkey-кнопок.

**Шаги:**
1. Изменить порядок кнопок (стрелки ↑↓ или перетаскивание) и сохранить.
2. Обновить страницу или снова открыть раздел.

**Ожидаемый результат:** Порядок сохраняется и отображается корректно. API: `PUT /api/activities/hotkeys/reorder` с `{"hotkey_ids": [id2, id1, ...]}` возвращает обновлённый список по `order`.

---

### TC-ACT-05 — Удаление hotkey-кнопки

**Предусловия:** Есть хотя бы одна запись в быстрых кнопках.

**Шаги:**
1. Удалить одну hotkey-кнопку (меню ⋮ → Удалить или аналог).
2. Подтвердить при необходимости.

**Ожидаемый результат:** Кнопка исчезает из списка; в боте эта кнопка больше не отображается. API: `DELETE /api/activities/hotkeys/{hotkey_id}` — 204.

---

### TC-ACT-06 — Удаление активности

**Предусловия:** Есть активность, не используемая в элементах плана (или приложение разрешает удаление с предупреждением).

**Шаги:**
1. Удалить активность из каталога активностей.
2. Подтвердить при необходимости.

**Ожидаемый результат:** Активность удаляется из списка. Если у активности была hotkey-кнопка — она также перестаёт быть доступна. API: `DELETE /api/activities/{activity_id}` — 204.

---

## 4. Веб: настройки пушей

### TC-PUSH-01 — Включение/выключение пушей (если реализовано в UI)

**Предусловия:** В веб-приложении есть страница или блок «Уведомления в Telegram» / «Настройки пушей».

**Шаги:**
1. Открыть раздел уведомлений.
2. Переключить «Пуши по расписанию» в положение Вкл/Выкл и сохранить.
3. Обновить страницу и проверить, что значение сохранилось.

**Ожидаемый результат:** Состояние переключателя сохраняется; при Выкл пуши не отправляются (если бэкенд это поддерживает).

**Примечания:** Если в MVP нет отдельной страницы или API для вкл/выкл пушей — зафиксировать «N/A» или «Не реализовано».

---

### TC-PUSH-02 — Тихие часы (если реализовано в UI)

**Предусловия:** В UI есть поля «Тихие часы» (время «с» и «до» в локальном времени).

**Шаги:**
1. Задать тихие часы (например, с 22:00 до 07:00) и сохранить.
2. Проверить, что в заданный интервал пуши не приходят (при наличии планировщика и элемента в этом окне).

**Ожидаемый результат:** Настройки сохраняются; в указанный интервал пуши не отправляются.

**Примечания:** Если тихие часы не реализованы — кейс пропустить.

---

## 5. Веб: отчёт за день

### TC-REPORT-01 — Открытие отчёта и выбор даты

**Предусловия:** Пользователь авторизован. Есть шаблон с элементами плана хотя бы на один день.

**Шаги:**
1. Открыть раздел «Отчёт за день».
2. Выбрать дату (date picker или стрелки «предыдущий/следующий день»), например сегодня или день с запланированными элементами.

**Ожидаемый результат:** Отображается отчёт за выбранную дату: блок «Запланировано» (время + название + тип) и блок «Факт» (пустой или с ответами). API: `GET /api/report/daily?date=YYYY-MM-DD` с заголовком `X-Web-Session-Id` возвращает `planned`, `answers`, `intervals`.

---

### TC-REPORT-02 — Отображение запланированного и фактов в отчёте

**Предусловия:** Выбрана дата, на которую в расписании есть дела/события. Опционально: на часть из них уже были ответы в боте.

**Шаги:**
1. Открыть отчёт за эту дату.
2. Сверить список «Запланировано» с элементами шаблона для выбранного дня недели.
3. Если были ответы на пуши — проверить, что в «Факт» отображаются статусы и время ответа.

**Ожидаемый результат:** Запланированные элементы соответствуют шаблону и дням недели. Факты (Сделал/Не сделал/Пропустить, Начал/Не начал, Закончил/Пропустил) и время отображаются корректно.

---

### TC-REPORT-03 — Интервалы (hotkey-сессии) в отчёте

**Предусловия:** За выбранный день были запущены и остановлены hotkey-сессии (через бота).

**Шаги:**
1. Открыть отчёт за этот день.
2. Найти блок «Интервалы» (или аналог).

**Ожидаемый результат:** Отображаются сессии по hotkey-активностям: название активности, начало, конец, длительность. API: в ответе `GET /api/report/daily?date=...` в `intervals` есть записи с `activity_name`, `started_at`, `ended_at`, `duration_seconds`.

## 6. Бот: пуши

### TC-BOT-PUSH-01 — Пуш «Дело»: кнопка «Сделал»

**Предусловия:** В расписании есть дело на время, близкое к текущему (или время теста подстроено). Планировщик пушей запущен; пуш за этот элемент уже отправлен в чат.

**Шаги:**
1. В боте найти сообщение вида «Сделал «<название>»?» с кнопками Сделал / Не сделал / Пропустить.
2. Нажать «Сделал».

**Ожидаемый результат:** Появляется ответ (всплывающее уведомление или редактирование сообщения): «Учтено». Кнопки убираются или сообщение меняется. В отчёте за день по этому элементу факт «Сделал» с временем ответа.

---

### TC-BOT-PUSH-02 — Пуш «Дело»: кнопки «Не сделал» и «Пропустить»

**Предусловия:** Есть пуш по делу (или имитировать разными делами).

**Шаги:**
1. Нажать «Не сделал» в одном сообщении.
2. В другом пуш-сообщении (другое дело) нажать «Пропустить».

**Ожидаемый результат:** В обоих случаях — «Учтено», кнопки обрабатываются. В отчёте — соответствующие статусы (Не сделал, Пропустить).

---

### TC-BOT-PUSH-03 — Пуш «Событие»: «Начал» / «Не начал»

**Предусловия:** В расписании событие; отправлен пуш «Начал «<название>» (время–время)?» с кнопками Начал / Не начал.

**Шаги:**
1. Нажать «Начал».
2. (Отдельно) Для другого пуш-сообщения события нажать «Не начал».

**Ожидаемый результат:** Сообщение обновляется, «Учтено». В отчёте за день — факты event_started и event_not_started соответственно.

---

### TC-BOT-PUSH-04 — Пуш «Событие»: «Закончил» / «Пропустил»

**Предусловия:** Отправлен пуш «Закончил «<название>»?» с кнопками Закончил / Пропустил.

**Шаги:**
1. Нажать «Закончил» в одном сообщении.
2. В другом — «Пропустил».

**Ожидаемый результат:** Оба ответа учтены, отображается «Учтено». В отчёте — event_ended и event_skipped.

---

### TC-BOT-PUSH-05 — Идемпотентность: повторное нажатие на кнопку пуша

**Предусловия:** По одному и тому же пушу уже нажата одна из кнопок (например, «Сделал») и сообщение показывает «Учтено».

**Шаги:**
1. Повторно нажать ту же кнопку (или оставшуюся кнопку того же сообщения, если кнопки ещё видны).
2. При повторной отправке callback (например, повторное нажатие «Сделал») — проверить ответ бота.

**Ожидаемый результат:** При повторном нажатии бот отвечает «Уже учтено» (или эквивалент), дубликат записи в БД не создаётся. Сообщение может обновиться на «Уже учтено» (по реализации).

## 7. Бот: hotkey и активные сессии

### TC-BOT-HK-01 — Отображение hotkey-кнопок в боте

**Предусловия:** Пользователь привязан. В веб настроены минимум одна hotkey-активность и одна быстрая кнопка (подпись, порядок). В боте реализован способ показа клавиатуры с быстрыми кнопками (например, команда или сообщение после привязки).

**Шаги:**
1. Открыть бота и получить сообщение с быстрыми кнопками (если есть отдельная команда или автоматическая отправка — выполнить её).
2. Убедиться, что отображаются кнопки с подписями, заданными на сайте.

**Ожидаемый результат:** В боте видны inline-кнопки (или reply-кнопки по реализации) с подписями из раздела «Быстрые кнопки» в том же порядке. Нажатие кнопки не приводит к ошибке (обрабатывается хендлером).

**Примечания:** Если в текущей версии клавиатура нигде не показывается — зафиксировать FAIL или «Нет способа получить кнопки».

---

### TC-BOT-HK-02 — Старт сессии по hotkey-кнопке

**Предусловия:** В боте доступны hotkey-кнопки. По данной активности нет активной сессии.

**Шаги:**
1. Нажать кнопку активности (например, «YouTube»).

**Ожидаемый результат:** Бот отвечает сообщением вида «Сессия «YouTube» начата.» (или «Запущено: YouTube…»). В списке активных сессий (/active или «Что сейчас идёт») появляется эта активность.

---

### TC-BOT-HK-03 — Остановка сессии повторным нажатием hotkey

**Предусловия:** Сессия по одной из hotkey-активностей уже запущена (TC-BOT-HK-02).

**Шаги:**
1. Нажать ту же hotkey-кнопку ещё раз.

**Ожидаемый результат:** Бот отвечает сообщением вида «Сессия «YouTube» закончена (N мин).» Длительность отображается корректно. В списке активных сессий эта активность исчезает. В отчёте за день появляется интервал с длительностью.

---

### TC-BOT-HK-04 — Команда /active или кнопка «Что сейчас идёт»

**Предусловия:** Пользователь привязан. Запущена минимум одна hotkey-сессия (или ни одной).

**Шаги:**
1. Отправить команду `/active` или нажать кнопку «Что сейчас идёт» (если она отображается).
2. При наличии активных сессий — проверить текст и кнопки «Закончить».

**Ожидаемый результат:** Если сессий нет: сообщение «Нет активных сессий. Нажми кнопку активности, чтобы начать трекать.» Если есть: список вида «Сейчас идёт: YouTube с 14:30; Работа с 09:00» и для каждой — inline-кнопка «Закончить <название>».

---

### TC-BOT-HK-05 — Завершение сессии через «Закончить»

**Предусловия:** Есть активная сессия; открыт список по /active или «Что сейчас идёт».

**Шаги:**
1. Нажать кнопку «Закончить <название>» у одной из активных сессий.

**Ожидаемый результат:** Сообщение обновляется: «Закончено» (или аналог). Всплывающее уведомление «Закончено». Сессия завершена, в отчёте за день записан интервал с длительностью.

---

### TC-BOT-HK-06 — Идемпотентность «Закончить»

**Предусловия:** Сессия уже завершена (нажата «Закончить» или повторно hotkey-кнопка), но сообщение со списком активных ещё отображает кнопку (или пользователь повторно нажимает «Закончить» по тому же session_id).

**Шаги:**
1. Повторно нажать «Закончить» для уже завершённой сессии (если кнопка осталась или воспроизвести через дубликат callback).

**Ожидаемый результат:** Бот отвечает «Уже закончено», дубликат записи session_end не создаётся. Обработка идемпотентна.

---

### TC-BOT-HK-07 — Две активности одновременно (инвариант: одна сессия на активность)

**Предусловия:** Настроены две разные hotkey-активности (две кнопки).

**Шаги:**
1. Нажать первую кнопку (старт сессии A).
2. Не останавливая A, нажать вторую кнопку (старт сессии B).
3. Проверить /active: должны быть обе сессии.
4. Завершить сначала одну, потом другую (кнопкой «Закончить» или повторным нажатием hotkey).

**Ожидаемый результат:** Одновременно активны две сессии (по разным активностям). По одной активности — только одна активная сессия. После завершения обеих в отчёте два интервала.

## 8. Связка веб–бот

### TC-E2E-01 — Настроил расписание и hotkey — в боте есть кнопки и пуши

**Предусловия:** Пользователь привязан. Планировщик пушей и бот запущены.

**Шаги:**
1. На сайте: создать шаблон, добавить дело и событие на сегодня (или на день недели и время, близкое к текущему). Добавить две hotkey-активности и вывести их в «Быстрые кнопки».
2. В боте: дождаться пушей по расписанию (или сдвинуть время в расписании для теста). Получить сообщение с hotkey-кнопками (если реализовано).
3. Ответить на пуш (дело или событие). Запустить и остановить одну hotkey-сессию.

**Ожидаемый результат:** Пуши приходят в заданное время с правильными текстами и кнопками. Hotkey-кнопки отображаются и работают (старт/стоп). Ответы на пуши и сессии сохраняются.

---

### TC-E2E-02 — Данные в отчёте после действий в боте

**Предусловия:** Выполнены ответы на пуши и одна hotkey-сессия за выбранный день (TC-E2E-01 или вручную).

**Шаги:**
1. Открыть на сайте раздел «Отчёт за день» и выбрать эту дату.
2. Проверить блоки «Запланировано», «Факт» и «Интервалы».

**Ожидаемый результат:** В «Факт» отображаются ответы на пуши (время и статус). В «Интервалы» — сессия по hotkey с названием активности и длительностью. Часовой пояс пользователя учтён (дата и время в отчёте соответствуют локальному времени).

## Прочие замечания

- **callback_data 64 байта:** Специально не проверять длину в байтах; достаточно убедиться, что все кнопки (пуши, hotkey, «Закончить») срабатывают без ошибок от Telegram.
- **Авторизация API:** Все эндпоинты, кроме `POST/GET /api/auth/link-code` и `GET /api/auth/link-status`, требуют заголовка `X-Web-Session-Id` с идентификатором привязанной сессии; иначе 401.
- При **FAIL** указывать: ID кейса, шаг, фактический результат и при необходимости версию/окружение.
