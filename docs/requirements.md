# Требования и граничные случаи

## 1. Функциональные требования (MVP)

### 1.1 Авторизация и связка
- Связать веб-сессию с пользователем Telegram через **одноразовый код** (TTL), подтверждаемый в боте (`/start <code>`).
- Код: только символы, допустимые в deep link (A–Z, a–z, 0–9, `_`, `-`), длина в пределах лимита start-параметра (до 512 символов).
- Защита от повторного использования: после успешного `consume_link_code` код помечается использованным и больше не принимается.
- После привязки сайт должен получить подтверждение (polling или redirect/обновление страницы) и открыть настройки.

### 1.2 Управление расписанием
- CRUD шаблона расписания (в MVP — один шаблон на пользователя).
- Элементы плана: **дело (TaskItem)** — одна точка времени; **событие (EventItem)** — временной блок (начало и конец/длительность).
- Повторение: минимум по дням недели (например, пн–пт / сб–вс).
- Учёт часового пояса пользователя: все времена в расписании задаются и отображаются в локальном времени; хранение и планирование — в UTC с использованием `timezone` пользователя.

### 1.3 Пуши в Telegram
- Планировщик по расписанию формирует и отправляет сообщения в заданное `planned_at` (в UTC с учётом timezone пользователя).
- **Дело**: одно сообщение с текстом вида «Сделал [название]?» и кнопками: Сделал / Не сделал / Пропустить.
- **Событие**: два типа пушей — «Начал [название]?» (Начал / Не начал) и «Закончил [название]?» (Закончил / Пропустил), отправляемые в плановое время старта и конца.
- На каждый ответ сохраняется `responded_at` (момент нажатия кнопки) и тип ответа (done, not_done, skipped для дел; started, not_started, ended, skipped для событий).

### 1.4 Hotkey-кнопки
- Пользователь на сайте настраивает список hotkey-кнопок (привязка к активностям, подпись, порядок).
- В боте отображается клавиатура/меню с этими кнопками.
- Нажатие кнопки: если сессии по этой активности нет — создаётся сессия (`started_at`); если есть — сессия завершается (`ended_at`). Альтернатива: отдельные кнопки «Начать»/«Закончить» на уровне реализации.
- Инвариант: у пользователя не более одной активной сессии на одну активность.

### 1.5 Список незавершённых активностей (в боте)
- По команде `/active` или кнопке «Что сейчас идёт» бот возвращает список активных сессий: название активности, время старта, опционально — длительность «уже идёт».
- Для каждой позиции — кнопка «Закончить» для завершения сессии и записи `ended_at`.

### 1.6 Просмотр результатов (веб)
- Отчёт за выбранный день: список запланированных элементов (дела и события), факты ответов (время ответа, статус), длительности интервалов по hotkey-сессиям.

## 2. Нефункциональные требования (MVP)

- **Надёжность**: обработка ответов на пуши и нажатий hotkey идемпотентна — повторная отправка того же callback не создаёт дубликатов записей и не меняет состояние некорректно.
- **Время**: все метки времени в БД хранятся в UTC; у пользователя хранится `timezone` (например, IANA: `Europe/Moscow`).
- **Безопасность**: минимум персональных данных; коды привязки с TTL и одноразовым использованием; при использовании webhook — проверка источника/подписи по рекомендациям Telegram.
- **Наблюдаемость**: структурированное логирование (например, JSON), при отправке пуша и при обработке ответа — общий корреляционный id (или `notification_id` / `idempotency_key`) для трассировки.

## 3. Граничные случаи и оговорки

### 3.1 Время и часовые пояса
- Переход на летнее/зимнее время: использовать IANA timezone (например, `Europe/Moscow`), расчёт локального времени и UTC вести через библиотеку (например, `zoneinfo` в Python).
- Если у пользователя не задан timezone: использовать значение по умолчанию (например, UTC) и явно предлагать настроить на сайте.

### 3.2 Повторы и расписание
- Один и тот же элемент плана (по id) в один день не должен порождать два одинаковых пуш-сообщения с одним и тем же idempotency_key.
- Если у элемента указано несколько дней недели, для каждой календарной даты генерируется своя «плановая» запись с одной датой и одним ключом идемпотентности.

### 3.3 Идемпотентность
- Callback-кнопки пушей содержат в `callback_data` идентификатор уведомления (или составной idempotency_key). При обработке callback:
  - Проверять, не был ли уже записан ответ по этому ключу.
  - Если ответ уже есть — не создавать новую запись, при необходимости обновить сообщение в боте («Уже учтено») и вернуть успех.
- Telegram ограничивает `callback_data` **64 байтами**. Использовать короткие префиксы и id (например, числовые или короткие строки), чтобы уложиться в лимит. При необходимости хранить маппинг «короткий id → notification_id» в кэше/БД.

### 3.4 Ошибки и лимиты Telegram
- **Ответ на callback**: обязательно вызывать `answer_callback_query`, иначе у пользователя может висеть индикатор загрузки.
- **Редактирование сообщения**: при повторном нажатии на ту же кнопку текст/разметка могут не измениться — API вернёт «message is not modified». Обрабатывать это как успех (идемпотентность), не логировать как ошибку.
- **Сетевые ошибки при отправке**: планировщик должен иметь ретраи с экспоненциальной задержкой; после исчерпания — логировать и при необходимости помечать уведомление как «не доставлено» для последующей ручной проверки или повторной отправки.
- **Блокировка бота пользователем**: при отправке сообщения Telegram может вернуть ошибку (например, bot was blocked by user). Не считать это критичной ошибкой приложения; логировать и не ретраить бесконечно.

### 3.5 Hotkey-сессии
- Если пользователь нажал «Закончить» для уже завершённой сессии (например, по дубликату callback): операция идемпотентна — возвращать успех, не создавать новую запись.
- Одновременно может быть несколько активных сессий у одного пользователя по **разным** активностям; по одной активности — только одна активная сессия.

### 3.6 Истечение кода привязки
- Если пользователь открывает бота с истёкшим кодом: сообщить «Код устарел. Запросите новый код на сайте.» и не создавать/не обновлять привязку.
- Если код уже использован: сообщить «Этот код уже использован.» и не менять привязку.
