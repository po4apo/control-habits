# Руководство по реализации Control Habits (для агентов и разработки в Cursor)

Краткий контекст для агентов и разработчиков: как декомпозировать задачи, в каком порядке делать и как организовать работу в Cursor (в т.ч. несколько агентов).

## Источники истины

- **Спека и план**: `plan.md` (цели, пользовательский путь, компоненты, стек).
- **Документация**: папка `docs/` — полное описание продукта и системы:
  - `docs/index.md` — оглавление.
  - `docs/requirements.md` — функциональные/нефункциональные требования, граничные случаи.
  - `docs/domain-model.md` — сущности, состояния событий.
  - `docs/system-design.md` — архитектура, потоки, БД, планировщик.
  - `docs/modules.md` — **разбиение на модули с интерфейсами и рекомендуемым порядком реализации**.
  - `docs/ui-telegram.md`, `docs/ui-web.md` — схемы UI.
  - **`docs/prompts.md`** — готовые промпты на каждую подзадачу (копировать в чат или под-агента).

При реализации всегда опираться на эти файлы; при расхождении — уточнять по ним.

## Декомпозиция задач

Задачи разбиты по **модулям** из `docs/modules.md`. Каждый модуль — отдельная зона ответственности с чёткими входами/выходами.

### Эпики (модули) в порядке реализации

| № | Модуль | Кратко | Задачи (примеры) |
|---|--------|--------|------------------|
| 1 | **storage** | Модели БД, миграции, репозитории | SQLAlchemy-модели, Alembic, UsersRepo, LinkCodesRepo, ScheduleRepo, ActivityRepo, HotkeysRepo, NotificationsRepo, LogsRepo, SessionsRepo |
| 2 | **auth_linking** | Код привязки сайт ↔ Telegram | create_link_code, consume_link_code, эндпоинты API, интеграция с ботом `/start <code>` |
| 3 | **schedule_model** | Развёртка шаблона на дату | DTO (TaskItem, EventItem, PlannedItem), expand_template(user_id, date) с учётом timezone |
| 4 | **planning_engine** | Что и когда пушить | build_notification_jobs, запись в notifications с idempotency_key |
| 5 | **bot_messages** | Тексты и клавиатуры для Telegram | build_task_prompt, build_event_start/end_prompt, build_hotkeys_keyboard, build_active_sessions_message, build_finish_buttons (лимит callback_data 64 байта) |
| 6 | **hotkey_sessions** | Старт/стоп/список сессий | start_session, stop_session, list_active_sessions, инвариант одна активная сессия на (user_id, activity_id) |
| 7 | **bot_handlers** | Обработка апдейтов бота | Роутинг callback, пуши, hotkey, /active, /start; идемпотентность ответов |
| 8 | **reporting** | Отчёт за день | get_daily_report(user_id, date) — planned, answers, durations |
| 9 | **scheduler** | Отправка пушей по расписанию | APScheduler или цикл: выборка pending notifications, отправка через Bot API, mark_sent, ретраи |

Дополнительно (сквозные):

- **Web API**: FastAPI-эндпоинты для CRUD расписания, активностей, hotkeys, отчёта, создания кода привязки и проверки статуса.
- **Web UI**: страницы по схемам из `docs/ui-web.md` (вход, редактор расписания, настройки пушей/hotkeys, отчёт).

Декомпозиция внутри модуля: разбивать на подзадачи по интерфейсам из `docs/modules.md` (например, по одному репозиторию или по одной функции).

## Организация работы в Cursor

### 1. Правила и контекст

- В `.cursor/rules/` лежит правило с контекстом проекта (стек, домен, ссылки на docs). Оно подтягивается при работе с релевантными файлами.
- В чате можно явно ссылаться на документы: «реализуй по `docs/modules.md` модуль storage» или «по `docs/requirements.md` п. 3.3 сделай идемпотентность».

### 2. plan.md и todo

- В `plan.md` уже есть блок с `todos` (спека). Для реализации удобно завести отдельный список задач (например, в Cursor Todo или в том же `plan.md`) по эпикам выше.
- Формулировать задачи так: «Реализовать модуль X: интерфейсы A, B из docs/modules.md».

### 3. Один агент — один фокус

- Для одной сессии лучше брать один эпик или одну подзадачу (например, «все репозитории storage» или «только auth_linking»). Так агент не теряет контекст и не смешивает слои.

### 4. Несколько агентов (mcp_task)

В Cursor можно запускать **под-агентов** через инструмент `mcp_task` для параллельной или специализированной работы:

- **Когда запускать несколько агентов**:
  - **Параллельно**: независимые подзадачи (например, «напиши тесты к модулю X» и «добавь миграцию для таблицы Y»), когда одна не зависит от другой.
  - **Отдельный исследовательский агент**: разобрать большой кусок кода или документации (например, «по docs и коду перечисли все места, где нужна идемпотентность») — тип `explore` или `generalPurpose`.
  - **Терминал/скрипты**: запуск миграций, тестов, линтеров — тип `shell`.

- **Типы под-агентов**:
  - **explore**: быстрый обзор кодовой базы, поиск по паттернам, ответы «где что лежит».
  - **generalPurpose**: поиск, исследование, многошаговые задачи (например, «реализуй по modules.md репозитории из storage»).
  - **shell**: выполнение команд (pytest, alembic, форматеры).

- **Как не дублировать работу**:
  - Чётко указывать в описании задачи границы: «только модели и миграции, без API».
  - Сначала реализовать зависимости (например, storage), потом модули, которые от него зависят (auth_linking, schedule_model и т.д.).
  - После работы под-агента проверять диффы и при необходимости дорабатывать в основной сессии.

Пример формулировки для под-агента:  
«В проекте control-habits по документу docs/modules.md реализуй модуль storage: SQLAlchemy-модели для всех таблиц из docs/system-design.md (раздел 4), миграция Alembic, интерфейсы репозиториев из modules.md (раздел 8). Стек: Python, SQLAlchemy 2.x, Alembic, PostgreSQL. Код в директории src/ (или уточни структуру). Не реализуй бизнес-логику и API — только слой хранилища.»

## Рекомендуемый рабочий поток

1. **Старт**: прочитать `docs/modules.md` и `docs/system-design.md`; при необходимости `docs/domain-model.md` и `docs/requirements.md`.
2. **Выбрать эпик**: по порядку 1→9; либо взять один модуль и разбить его на подзадачи в Todo.
3. **Одна сессия агента**: одна подзадача или один модуль; в промпте указывать «по docs/modules.md» и при необходимости «по docs/requirements.md п. 3.3».
4. **Параллельные агенты**: только для независимых подзадач (например, тесты к уже готовому модулю + миграция для следующего).
5. **Интеграция**: после модулей bot_handlers и scheduler — прогон сценариев по plan.md (онбординг, пуш, ответ, hotkey, отчёт).
6. **Ведение плана**: обновлять plan.md или отдельный implementation-plan: отмечать выполненные модули, добавлять следующие задачи.

## Структура кода (ориентир)

- Хранить код приложения в одной корневой папке (например, `src/` или пакеты в корне).
- Модули — пакеты или подпапки: `storage/`, `auth_linking/`, `schedule_model/`, `planning_engine/`, `bot_messages/`, `hotkey_sessions/`, `bot_handlers/`, `reporting/`, плюс `api/`, `scheduler/`, `web_ui/` при необходимости.
- Конфиг: pydantic-settings, переменные окружения.
- Тесты: рядом с модулями или в `tests/` с зеркальной структурой; моки соседних слоёв для изоляции.

Эта структура может быть уточнена в `docs/system-design.md` или в отдельном файле по соглашению команды.
